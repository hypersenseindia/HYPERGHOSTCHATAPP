What 3 Layers Actually Means.

Your message doesn't get encrypted once. It gets encrypted, then the key gets encrypted, then THAT key gets encrypted. Three separate AES-256-GCM operations. Three independent keys. Break one — you still see nothing.

What The Server Sees

The Firebase database stores exactly two fields per message: msg and key. Both are base64-encoded ciphertext. The msg field is the triple-encrypted message. The key field is the double-encrypted per-message key. Neither can be decrypted without your device's crypto context. There is no plaintext. There is no metadata about content. There is no fallback.

What Your Device Does

Every message triggers three sequential AES-256-GCM operations: (1) encrypt plaintext with a fresh random Key1, (2) encrypt Key1 using a key derived from the sender's UID via SHA-256, (3) encrypt that result using the recipient's public key. The server receives two opaque blobs. To reverse this, you need the recipient's public key AND the sender's UID AND the correct AES-GCM context — all on the same device.

HX-E2E Architecture

The 3 Layers,
Broken Down.

Each layer uses independent AES-256-GCM encryption. Breaking any single layer reveals nothing useful. You need all three, in the correct order, on the correct device.

Layer 1

Message Encryption

Plaintext is encrypted with a fresh random 256-bit Key1 using AES-GCM with a random 12-byte IV.

Key1 → AES-GCM → Ciphertext
Layer 2

UID-Locked Key

Key1 is encrypted using a key derived from the sender's UID via SHA-256. Binds the key to the sender's identity.

SHA-256(senderUID) → AES-GCM → LockedKey1
Layer 3

Public Key Wrap

The locked Key1 is encrypted with the recipient's public key. Only the recipient can begin the decryption chain.

recipientPubKey → AES-GCM → FinalBlob

Decryption Chain (Recipient)

Step 1: Unwrap FinalBlob with your public key → get LockedKey1.

Step 2: Unlock LockedKey1 using sender's UID (SHA-256 derived) → get Key1.

Step 3: Decrypt ciphertext with Key1 → get plaintext.

All three steps must succeed. Miss one = unreadable garbage.

What Each Broken Layer Reveals

Break Layer 3 only: You get LockedKey1 — still encrypted with the sender's UID. Useless.

Break Layer 2 only: You get Key1 — but you don't have the encrypted message (it's inside the Layer 3 blob). Useless.

Break Layer 1 only: You get ciphertext — but Key1 is locked behind two more layers. Useless.

Each layer is independently useless without the others.

Technical Breakdown

How The 3-Layer
Encryption Actually Works

Step by step. No hand-waving. This is the exact flow when you send a message.

1
Layer 1 — Generate Fresh Key

Per-Message Random Key

A random 256-bit AES-GCM key (Key1) is generated via crypto.subtle.generateKey(). This key exists only in memory for this one message. It will never be reused.

2
Layer 1 — Encrypt Message

Plaintext → Ciphertext

Your message is encrypted with Key1 using AES-GCM with a random 12-byte IV. Output: EncryptedMsg. This is what carries your actual words — but it's unreadable without Key1.

3
Layer 2 — Derive UID Key

SHA-256(senderUID) → UID Key

Your Firebase UID is hashed with SHA-256 to produce a deterministic 256-bit key. This key is derived from your identity — anyone who knows your UID can derive the same key. This is intentional: it binds the message key to you.

4
Layer 2 — Lock Key1

Key1 → LockedKey

Key1 (from step 1) is exported to base64, then encrypted with the UID-derived key using AES-GCM. Output: LockedKey. Even if someone gets this, they need the sender's UID to unlock it.

5
Layer 3 — Public Key Wrap

LockedKey → FinalBlob

The LockedKey is encrypted using the recipient's public key (stored at users/{uid}/pubKey). Output: FinalKey. Only the recipient can begin the unwrapping chain.

6
Transmit

Send To Firebase

Two fields are sent: msg: EncryptedMsg and key: FinalKey. The server sees two base64 blobs. That's all it will ever see. After delivery, the server copy is deleted.

3-Layer Implementation
Secrets Revealed

Only visible to those who read the entire page. Here's what's actually happening under the hood of each layer.

Trick #01

Why UID-Locking Actually Works

Layer 2 uses SHA-256(senderUID) as the key to encrypt Key1. The sender's UID is public information — it's in Firebase at users/{uid}. So isn't this layer useless? No. The point isn't to hide the key — it's to bind Key1 to the sender's identity. Without knowing who sent the message, you can't unlock Layer 2 even if you break Layer 3. It's an identity verification layer, not a secrecy layer. Combined with Layers 1 and 3, it means: you need the recipient's key AND the sender's UID AND the per-message key. Three different types of knowledge.

Trick #02

3 Layers vs Double Ratchet

Signal's Double Ratchet provides forward secrecy by deriving each key from the previous one via DH key exchange. Our approach is different: we generate a completely random key per message (no chain), then add two more encryption layers on top. Tradeoff: Signal's ratchet means compromising one key doesn't reveal past OR future messages. Our 3-layer system means compromising one message's key chain reveals only that one message (because each Key1 is independent). We lose forward secrecy but gain per-message isolation — a compromised message N's full 3-layer chain gives you message N only. Message N-1 uses a completely unrelated Key1.

Trick #03

IV Safety Across 3 Layers

Each of the 3 AES-GCM calls uses crypto.getRandomValues(new Uint8Array(12)) for its IV. That's 3 independent random IVs per message. In Layer 1, even if the IV collides (impossible at 96-bit space), the Key1 is different for every message. In Layer 2, the UID-derived key is the same per sender, but the IV is random — safe. In Layer 3, the recipient's key is the same, but IV is random and the input (LockedKey) is different every time. All three layers are IV-safe independently.

Trick #04

The Delete-After-Read Pattern

After a message is received and decrypted, the Firebase copy is immediately deleted via db.ref('chats/'+snap.key).remove(). The encrypted blob exists on the server for only milliseconds — the time between child_added firing and the delete completing. This means: even if someone gains access to your Firebase at a random moment, most messages won't even be there. Only the local IndexedDB copy (decrypted) persists on your device.

Result: Server contains messages only during transit. At rest, the server is nearly empty.
Trick #05

Why Public Key In Firebase Is Acceptable

The recipient's public key (users/{uid}/pubKey) is stored in Firebase in base64 format. This is by design — it's a public key in the asymmetric sense (even though we use symmetric AES-GCM). Anyone can read it. But having the public key alone doesn't help: you'd also need to break Layer 3 (which requires the public key — circular), then break Layer 2 (which requires the sender's UID — public but you need to know who sent it), then break Layer 1 (which requires Key1 — locked behind the other two). The public key is the entry point to decryption, not a vulnerability.

Trick #06

The Actual Weakness (Honest)

Weakness 1: The public key in Firebase means the admin (us) CAN read your public key. Combined with any sender's UID (also public), the admin could theoretically decrypt Layer 3 and Layer 2 of any message — if they intercept it during the millisecond it exists on the server before delete-after-read removes it. In practice: nearly impossible without real-time listening.

Weakness 2: No forward secrecy. If your device is compromised while the app is open, all cached decrypted messages in IndexedDB are readable.

Weakness 3: The UID-derived key in Layer 2 is deterministic — same UID always produces the same key. This is a design choice for identity binding, not a bug.

Mitigation: For protection against the platform operator, use this alongside the knowledge that messages are deleted from the server almost instantly. For protection against device compromise, close the tab when not in use — IndexedDB data persists but crypto.subtle keys are purged from memory.
